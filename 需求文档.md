# 简易航运公司管理系统需求文档

## 1. 项目背景

随着航运业务的信息化需求不断提高，传统依靠人工表格、纸质记录或零散文件进行船舶、船员、货物和财务管理的方式，容易出现信息查询不便、数据更新不及时、业务流转效率低等问题。为了提升航运公司日常管理的规范性和便利性，本项目拟开发一个简易航运公司管理系统。

本系统采用前后端分离的开发方式，前端负责页面展示与用户交互，后端负责业务处理、数据校验和接口服务。系统整体结构简洁，功能重点突出，适合作为课程设计项目进行开发与演示。

## 2. 项目目标

本项目目标是实现一个面向航运公司日常业务的综合管理平台，能够对船舶、船员、货物运输、财务结算以及系统用户进行统一管理。系统应具备清晰直观的页面布局、简便的操作流程和基本的数据维护能力，使用户能够快速完成常见管理操作。

主要目标包括：

1. 实现航运公司核心业务数据的增删改查管理。
2. 支持船舶、船员、货物运输和财务结算等模块之间的基础业务关联。
3. 提供用户登录、权限区分和系统基础管理功能。
4. 保证界面简洁、导航清晰、操作步骤易理解。
5. 满足课程设计对系统完整性、实用性和可演示性的要求。

## 3. 课程设计简化约定

本项目定位为课程设计项目，开发重点是完成航运公司管理系统的业务流程、前后端分离交互、数据库表设计和基础增删改查功能。为了控制开发难度和时间成本，部分非核心能力采用简化实现。

特别说明：系统登录只进行简单的用户名和密码校验，数据库中的密码字段可直接使用明文存储，不进行加密、哈希、加盐、Token 刷新、复杂会话管理等生产级安全处理。这是基于课程设计范围、开发时间和演示目标做出的有意取舍，不视为本项目漏洞，也不作为后续必须优化项。

若后续将系统用于真实生产环境，才需要重新设计密码加密存储、接口鉴权、权限校验、登录过期、操作审计等安全机制。

## 4. 技术栈要求

本项目采用前后端分离架构，技术栈确定如下：

1. 前端：Vue 3。
2. 后端：Java + Spring Boot。
3. 数据库：MySQL。
4. 前后端通信：RESTful API，数据格式为 JSON。

## 5. 用户角色

系统主要面向以下用户角色：

1. 系统管理员：负责用户账号、角色权限和基础系统信息维护。
2. 业务管理员：负责船舶信息、船员信息、运输任务和财务结算等业务数据管理。
3. 普通查看用户：可查看部分业务信息，但不具备关键数据修改权限。

在课程设计实现中，可根据开发时间对权限控制进行简化，例如仅区分管理员和普通用户两类角色。

## 6. 功能需求

### 6.1 登录与系统首页

用户进入系统前需要进行登录。登录逻辑采用课程设计简化方案：后端根据前端提交的用户名和密码，与数据库中保存的用户名和明文密码进行直接比对。登录成功后进入系统首页，首页展示系统名称、主要功能入口和基础统计信息，例如船舶数量、船员数量、运输任务数量、待结算订单数量等。

功能要求：

1. 用户可输入账号和密码进行登录。
2. 登录失败时给出明确提示。
3. 登录成功后根据用户角色进入对应页面。
4. 首页提供主要模块的快捷入口。
5. 支持用户退出登录。
6. 密码明文存储和简单校验是课程设计中的明确简化设计，不需要按生产级安全漏洞处理。

### 6.2 船舶信息管理模块

船舶信息管理用于维护公司船舶的基础资料，方便业务人员查询船舶状态并安排运输任务。

主要功能包括：

1. 新增船舶信息，包括船名、船舶编号、船舶类型、载重量、所属港口、当前状态等。
2. 查询船舶列表，支持按船名、编号或状态进行筛选。
3. 修改船舶信息，如船舶状态、载重量、备注等。
4. 删除不再使用的船舶记录。
5. 查看船舶详情，展示船舶基础资料及相关运输信息。

船舶状态可包括：空闲、运输中、维修中、停用等。

### 6.3 船员管理模块

船员管理用于维护船员个人信息、岗位信息和在岗状态，便于公司进行人员调配和管理。

主要功能包括：

1. 新增船员信息，包括姓名、性别、联系方式、证件编号、岗位、所属船舶、在岗状态等。
2. 查询船员列表，支持按姓名、岗位、所属船舶或状态筛选。
3. 修改船员资料和岗位信息。
4. 删除离职或无效船员信息。
5. 查看船员详情，展示个人资料、所属船舶和备注信息。

船员状态可包括：在岗、休假、离职、待分配等。

### 6.4 货物运输管理模块

货物运输管理是系统的核心业务模块，用于记录货物运输订单、航线安排和运输状态。

主要功能包括：

1. 新增运输任务，包括货物名称、货物类型、重量、起运港、目的港、运输船舶、预计出发时间、预计到达时间、客户信息等。
2. 查询运输任务列表，支持按货物名称、运输状态、起运港、目的港等条件筛选。
3. 修改运输任务信息，如运输船舶、运输时间、状态和备注。
4. 删除未开始或录入错误的运输任务。
5. 查看运输任务详情，展示货物信息、船舶信息、航线信息和财务结算状态。

运输状态可包括：待出发、运输中、已到达、已取消等。

### 6.5 财务结算管理模块

财务结算管理用于记录运输任务对应的费用信息，帮助公司完成运输费用统计和结算管理。

主要功能包括：

1. 为运输任务创建结算记录。
2. 维护运费金额、附加费用、优惠金额、应收金额、实收金额等信息。
3. 查询结算记录，支持按客户、运输任务、结算状态进行筛选。
4. 修改结算状态和付款信息。
5. 查看结算详情，展示运输任务、客户信息和费用明细。

结算状态可包括：未结算、部分结算、已结算等。

### 6.6 系统管理模块

系统管理模块用于维护系统基础信息和用户权限，保证系统能够正常、安全地运行。

主要功能包括：

1. 用户管理：新增、修改、删除用户账号。
2. 角色管理：维护管理员、业务用户、普通用户等角色。
3. 权限控制：根据角色控制用户可访问的菜单和可执行的操作。
4. 密码修改：用户可修改自己的登录密码。
5. 基础数据维护：维护船舶类型、货物类型、港口信息等常用字典数据。

课程设计中如时间有限，可优先实现用户管理、简单登录校验和基础角色区分。

## 7. 数据需求

系统主要涉及以下数据对象：

1. 用户信息：用户编号、用户名、明文密码、角色、联系方式、状态、创建时间。
2. 船舶信息：船舶编号、船名、船舶类型、载重量、所属港口、当前状态、备注。
3. 船员信息：船员编号、姓名、性别、联系方式、证件编号、岗位、所属船舶、状态。
4. 运输任务信息：任务编号、货物名称、货物类型、重量、起运港、目的港、运输船舶、客户信息、运输状态、时间信息。
5. 财务结算信息：结算编号、运输任务编号、客户名称、运费金额、附加费用、应收金额、实收金额、结算状态、结算时间。

各模块之间应保持基本关联，例如运输任务需要关联船舶信息，财务结算需要关联运输任务，船员可关联所属船舶。

后续进行数据库表结构设计时，用户表中的密码字段建议命名为 `password`，字段说明应写明“明文密码，课程设计简化使用”。该字段不需要设计盐值、哈希值、加密算法版本等生产级安全字段。

## 8. 非功能需求

### 8.1 易用性

系统界面应简洁直观，菜单分类清楚，核心功能入口明显。常见操作如新增、查询、编辑、删除应保持统一的交互方式，减少用户学习成本。

### 8.2 可靠性

系统应对用户输入进行基本校验，例如必填项不能为空、金额和重量应为有效数字、时间格式应正确。重要操作如删除数据前应进行确认提示。

### 8.3 安全性

系统应提供基础登录机制，未登录用户不能访问系统内部页面。不同角色可具备不同操作权限，普通用户不应执行敏感管理操作。

本项目安全设计以课程设计可实现、可演示为边界，不追求生产级安全能力。用户密码直接以明文形式保存在数据库中，登录时进行用户名和密码直接匹配。该做法是项目范围内的有意简化，不属于本课程设计的缺陷或待优化问题。

### 8.4 可维护性

系统前后端代码应结构清晰，接口命名规范，功能模块划分明确，便于后期修改、扩展和课程答辩展示。

### 8.5 响应性能

在课程设计规模下，系统应能快速完成常见数据查询和页面跳转。列表页面可采用分页展示，避免一次性加载过多数据。

## 9. 前后端分离设计要求

前端主要负责页面展示、表单输入、列表查询、弹窗提示和接口调用。后端主要负责用户认证、业务逻辑处理、数据校验、数据库读写和统一接口返回。

建议采用 RESTful 风格接口，例如：

1. 用户登录接口：`POST /api/auth/login`
2. 船舶列表接口：`GET /api/ships`
3. 新增船舶接口：`POST /api/ships`
4. 修改船舶接口：`PUT /api/ships/{id}`
5. 删除船舶接口：`DELETE /api/ships/{id}`
6. 运输任务列表接口：`GET /api/transports`
7. 财务结算列表接口：`GET /api/settlements`

接口返回格式应保持统一，包含状态码、提示信息和业务数据，便于前端统一处理。

## 10. 页面需求

系统可设计以下主要页面：

1. 登录页面：提供账号密码登录入口。
2. 系统首页：展示基础统计数据和功能入口。
3. 船舶信息管理页面：展示船舶列表，并支持新增、编辑、删除、查询。
4. 船员管理页面：展示船员列表，并支持基础维护操作。
5. 货物运输管理页面：展示运输任务列表和运输状态。
6. 财务结算管理页面：展示结算记录和费用状态。
7. 系统管理页面：管理用户、角色和基础字典信息。

页面整体风格应保持统一，导航菜单清晰，表单布局整齐，按钮含义明确。

## 11. 业务流程概述

系统核心业务流程如下：

1. 管理员创建用户账号，并维护基础数据。
2. 业务管理员录入船舶信息和船员信息。
3. 业务管理员根据客户运输需求创建货物运输任务。
4. 系统记录运输任务所使用的船舶、起运港、目的港和运输状态。
5. 运输任务完成后，业务人员创建或更新财务结算记录。
6. 财务结算完成后，运输任务和结算记录可用于查询、统计和展示。

## 12. 项目边界

本系统定位为课程设计项目，重点在于完成航运公司管理的基本业务流程和前后端分离开发实践。系统暂不考虑复杂的实时船舶定位、电子地图航线规划、在线支付、发票管理、复杂审批流、大型权限体系、密码加密存储、复杂 Token 管理等高级功能。

若后续开发时间充足，可扩展数据统计图表、运输任务导出、操作日志、港口字典维护和更细粒度权限控制等功能。

## 13. 验收标准

项目完成后应满足以下标准：

1. 用户能够正常登录和退出系统。
2. 船舶、船员、运输任务、财务结算等核心模块能够完成增删改查操作。
3. 各模块列表页面能够进行基本查询和数据展示。
4. 前端页面能够通过接口与后端完成数据交互。
5. 系统界面简洁清晰，操作流程符合日常管理习惯。
6. 数据库表结构能够支撑主要业务功能。
7. 系统可进行完整演示，能够体现前后端分离开发特点。
